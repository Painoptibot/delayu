**Участники:** те же специалисты + начальник (на утверждение).  
**Цель:** согласовать экраны прототипа.

### E1. По каждому экрану — единый чек-лист

Для **каждого** из 8+ экранов прототипа:

| № | Вопрос |
|---|--------|
| E1.1 | Какие **поля на одном экране**, какие — на вкладках? |
| E1.2 | Какие **действия** на панели (создать, сохранить, печать, отправить на утверждение)? |
| E1.3 | Какие **фильтры** в реестрах обязательны? |
| E1.4 | Какие **цветовые статусы** (просрочено, на согласовании)? |
| E1.5 | Нужны ли **подсказки** по законодательству в интерфейсе? |
| E1.6 | **Мобильная версия** нужна или только desktop? |

### E2. Список экранов для подписи в протоколе

| № | Экран | Вопросы согласования |
|---|-------|---------------------|
| 1 | Рабочий стол | Виджеты: просрочки, новые заявления, обращения? |
| 2 | Реестр заявлений | Колонки, сортировка по умолчанию, экспорт? |
| 3 | Карточка заявления | Вкладки: заявитель, семья, документы, расчёт, история? |
| 4 | Расчёт дохода/имущества | Табличный ввод или мастер? |
| 5 | Учётное дело | Очерёдность, категория, основания? |
| 6 | Обращение | Таймер 30 дней, продление? |
| 7 | Отчёты | Параметры, предпросмотр, формат выгрузки? |
| 8 | Администрирование | Кто может создавать пользователей? |
| 9 | Дети-сироты (дело) | Пакет документов, запросы ведомств? |
| 10 | Договор | Связи, сканы, статусы? |
| 11 | МКД / помещение | Карточка объекта, план? |
| 12 | Проверка (контроль) | Предписания, нарушения? |

### E3. Роли и права 🔴

| № | Вопрос |
|---|--------|
| E3.1 | Подтвердите роли из ТЗ: админ, начальник, зам, специалисты, инспектор, отчётность. |
| E3.2 | Кто может **удалять** записи (или только «мягкое» удаление)? |
| E3.3 | Кто **утверждает** заключения и решения? |
| E3.4 | Нужен ли **доступ только чтение** для комиссии / руководства администрации? |
| E3.5 | Матрица «роль × модуль» — заполнить таблицу на визите. |

### E4. Уведомления и задачи 🟡

1. Внутренние **напоминания** (срок заявления, обращение) — кому, как?
2. E-mail / мессенджеры — нужны ли **вне системы**?
3. Лента задач на рабочем столе — нужна?

# Часть F. Визит 4 — MVP, закупка, приёмка (23–27.06.2026)

**Участники:** руководство **УЖВ** + **ИКТ** (закупка, приёмка, сервер) + при необходимости юрист.

### F1. Объём MVP и контракта 🔴

| № | Вопрос | Решение (заполнить) |
|---|--------|---------------------|
| F1.1 | Подтверждаете **перечень подсистем в MVP** (лист согласования)? | ☐ Да ☐ С изменениями |
| F1.2 | Какие функции **исключаем из первого контракта** явно? | |
| F1.3 | **Опытная эксплуатация** — сколько дней (14+)? | |
| F1.4 | Критерии **приёмки** — чек-лист из скольких пунктов? | |
| F1.5 | **Комиссия:** председатель ИКТ, представитель УЖВ (функционал)? | |
| F1.5а | Раздельные акты: **технический** (ИКТ) и **функциональный** (УЖВ) или один? | |

### F2. Закупка и реестр 🔴

1. Дата **публикации** извещения (целевая)?
2. Формулировка: «ПО из реестра» + **функциональные требования** — согласование с юристом.
3. Нужно ли упоминание **прототипа и протоколов** в приложении к ТЗ?
4. **Квалификационные требования** к участнику (опыт, пилот)?

### F3. Обучение и ввод в эксплуатацию 🟡

1. Сколько **групп обучения**, по сколько человек?
2. Нужны ли **видеоинструкции** или только очные занятия?
3. Кто **суперпользователи** на стороне УЖВ?

### F4. Риски и «что если» 🟡

1. Если **реестр задержится** — переносим закупку или контракт с отлагательным условием?
2. Если **нет сервера** к сентябрю — временный стенд исполнителя допустим?
3. План B при **проигрыше торгов** — кто продолжает развитие?
